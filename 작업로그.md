## 1. 문서 유지·갱신 규약

### 1.1 갱신이 반드시 필요한 시점
다음 중 하나라도 해당하면 **같은 턴 안에서** 로그 엔트리를 추가한다.

- 새 파일 생성·삭제
- 기존 파일의 의미 있는 수정 (오탈자 수정 등 자명한 것 제외)
- 사용자 의사결정 수신
- 계획·방향 전환
- Phase 전환 (예: Phase A → B)
- 스킬·에이전트 호출을 수반한 작업
- 사용자가 중요 맥락 · 고객 정보 · 용어 정의를 제공한 경우

### 1.2 엔트리 템플릿 (8 필드 필수)

```
### [연번]. [제목] (YYYY-MM-DD[, 세션 N])
- **맥락**: 직전 상태와 이 작업이 왜 나왔는지.
- **사용자 요청 (원문 요지)**: 사용자가 실제로 한 말 또는 짧은 직접 인용.
- **AI 수행**: 무엇을 · 어떤 도구·스킬·에이전트로 · 어떤 순서로.
- **판단 근거**: 왜 그 방식을 택했는가, 대안과 트레이드오프.
- **사용자 의사결정**: 사용자가 내린 결정 + 가능하면 원문 인용.
- **산출물**: 생성/수정된 파일·설정·메시지, 경로·라인 포함.
- **배운 점·재사용 포인트**: 다른 프로젝트로 이식 가능한 패턴.
- **다음 단계**: 이 엔트리 이후 열린 과제.
```

### 1.3 기재 원칙
- **사실은 정확히, 추정은 추정이라 표시.**
- **"왜" 를 반드시 남긴다** — "무엇" 보다 "왜" 가 이식성의 원천.
- **사용자 발언 직접 인용** 짧게라도 포함 — 요약은 의도 왜곡의 주범.
- **경로는 절대경로**, 라인 번호는 수정 시점 기준 명시.
- 한 엔트리가 길어지면 나눠서 두 엔트리로 — 한 엔트리 = 한 의사결정 단위.

### 1.4 다른 작업자 인계 시 읽는 순서
1. 섹션 2 (프로젝트 기본 정보 스냅샷) — 기준일 확인.
2. 섹션 3 (진행 연대기) 최근 5 엔트리 + 미결 연결 엔트리.
3. 섹션 5 (미결 항목) — 바로 착수할 것.
4. 섹션 4 (방법론 추출) — 일하기 전 한 번.
5. 필요 시 섹션 3 의 해당 엔트리 역추적.

## 2. 프로젝트 기본 정보 스냅샷

### 2.1 정체성
- 한국어 사업계획서 작성용 **콘텐츠 제작 워크스페이스** (소프트웨어 프로젝트 아님).
- 목표 문서: 정부지원 제조 AI 사업계획서 / 사업신청서 / 수행계획서 / 연구개발계획서.
- 언어: **한국어 문어체** (영어·구어체 금지).
- 출력 형식: Markdown 초안 → 사용자가 .hwp / .docx / .pptx 로 이식.

### 2.2 고객사 범위
- 주 타겟: **철강 및 부산·경남권 제조업체**.
- 규모: 대기업급 ~ 중소 (편차 큼).
- 실명 예: 화승, YCP특수강, 코리녹스, 코렌스, 대한제강, 한국철강.
- 업종: 철강·금속가공(연속주조·압연·열처리·표면처리) 중심 + 고무·폴리머 + 정밀가공.

### 2.3 작성 접근 방식 — "공통 자산 + 시나리오 특화"
1. AI 관련 공통 본문·삽화 (3대 트랙 기반) 축적.
2. 시나리오 정의.
3. 시나리오별 특화 문서·삽화 파생 → 재조합 가능한 모듈.

### 2.4 3대 트랙
- **Track 1**: 제조 AI 서두/본문 — 필요성, 데이터 유형, 피쳐·모델, 파이프라인.
- **Track 2**: MLOps 구축 및 지속적 개선.
- **Track 3**: LLM + RAG 공정 적용.
